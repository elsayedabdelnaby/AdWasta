# Research production multi-tenant SaaS gaps

## Gaps vs. Production Multi-Tenant B2B SaaS Requirements

**Authentication / Authorization / RBAC (control-plane API)** â€” The plan has zero auth phase: no login, session model, or token scheme anywhere in the API surface (`docs/implementation-plan.md` Phase 0â€“10), and design.md Â§24 explicitly punts "multi-user RBAC (single user per tenant OK for v1)" with no mention of even single-user login. Every endpoint (`POST /tenants`, `PATCH /tenants/:id/platforms/:platform`, credential save) is undefined w.r.t. who's allowed to call it â€” this is a live vulnerability class, not a v2 nicety, especially once this handles customer social/email credentials. Multi-tenant auth requires tenant-scoped identity from day one: "every authorization decision must be tenant-aware... you check 'is user an admin in this tenant?'" ([WorkOS](https://workos.com/blog/how-to-design-multi-tenant-rbac-saas)); B2B auth needs per-tenant login config and session hardening ([SuperTokens](https://supertokens.com/blog/secure-multi-tenant-auth)). Cheap fix for a small Node/Fastify team: **WorkOS AuthKit** (free to 1M MAU, has orgs/roles built in, SSO/SCIM pay-as-you-grow) or **Clerk Organizations** for fast RBAC + multi-tenant membership ([WorkOS vs Clerk](https://workos.com/blog/workos-vs-betterauth-vs-clerk)) â€” either beats hand-rolling JWT + RBAC middleware.

**Billing / Metering / Usage-Based Pricing** â€” Plan tracks `agent_traces.cost_usd` per tenant/day (Â§6, Â§18) but has no invoicing, plan tiers, quota enforcement tied to a subscription, or payment processor integration anywhere. For a multi-tenant SaaS charging real customers this is existential, not optional. Since 2025 Stripe requires every metered price to back to a Meter object with Meter Events ([Stripe metering guide](https://www.buildmvpfast.com/blog/stripe-metered-billing-implementation-guide-saas-2026)), and 74% of SaaS vendors now use usage-based models ([same source](https://www.buildmvpfast.com/blog/stripe-metered-billing-implementation-guide-saas-2026)). Recommended: **Stripe Billing** (meter events fed straight from your existing `agent_traces` cost rollup) or, if you want it self-hosted/open-source, **Lago** (Node.js SDK, consumption tracking + subscriptions) or **OpenMeter/Kong Metering** ([Lago](https://github.com/getlago/lago), [OpenMeter](https://openmeter.io/)).

**Rate Limiting (per-tenant + per-endpoint)** â€” Currently one bullet in Phase 10's checklist. A noisy tenant's research/LLM job storm or a runaway ReAct loop should not degrade every other tenant sharing the one Postgres/Redis instance â€” this needs design now, not a footnote. Production pattern: token bucket in Redis keyed `rl:{tenantId}:{endpointId}` with atomic Lua-script refill/consume ([Redis rate limiter guide](https://redis.io/tutorials/howtos/ratelimiting/), [multi-tenant design](https://medium.com/@khalilsayed/system-design-multi-tenant-rate-limiting-service-32c63ade5ec7)). Since the stack is already Fastify + ioredis, use **`@fastify/rate-limit`** with a custom `keyGenerator` returning `tenantId` and a Redis-backed store, tiered separately for LLM-cost endpoints vs. cheap reads ([`@fastify/rate-limit`](https://github.com/fastify/fastify-rate-limit)).

**Tenant Onboarding UX** â€” Plan has one API call (`POST /tenants/:id/onboard`, Zod-validated). No progressive disclosure, empty states, first-value checklist, or activation flow is designed, yet onboarding quality is the single biggest activation/retention lever in SaaS ("layer complexity over time... 3-5 checklist items tied directly to core value" â€” [Userpilot](https://userpilot.com/blog/onboarding-ux-best-practices/), [SaaSUI](https://www.saasui.design/blog/saas-onboarding-ux-examples)). Given four pillars and many toggles, a bare intake form will bury new tenants. Cheap pattern: a guided setup checklist component (e.g. **Userpilot** or a homegrown `onboarding_checklist` table driving the Phase-9 dashboard) sequencing profile â†’ platform connect â†’ first campaign run.

**Database Backup & PITR** â€” Not mentioned once despite Postgres being the sole system of record for every tenant's business data, credentials, and audit trail. "An untested backup is not a backup" ([Percona](https://www.percona.com/blog/postgresql-backup-strategy-enterprise-grade-environment/)); WAL-gap = data loss window ([pgBackRest guide](https://mydba.dev/blog/postgres-point-in-time-recovery)). Concrete: **pgBackRest** for base+incremental backups and WAL archiving to S3-compatible storage, or if using a managed provider (Neon/RDS), enable native PITR â€” either way, schedule quarterly restore drills.

**GDPR/CCPA Deletion & Export ("right to be forgotten")** â€” Design mentions "redact emails/phones in payload when possible" for DM content only (Â§10.1) â€” no tenant-level data export or deletion API, and crucially no procedure for purging PII from `agent_traces`, `system_events`, `intel_snapshots`, backups, or eval fixtures. Regulations require complete erasure within a defined window (commonly 30 days) across *all* systems including backups and queues ([Ketch](https://www.ketch.com/blog/posts/dsars-101-how-to-handle-data-deletion-requests), [Optimizely](https://support.optimizely.com/hc/en-us/articles/4410284226061)). Given DM contents and competitor-page scrapes are stored, this is a real gap. Build `POST /tenants/:id/export` (dump all tenant-scoped tables to signed S3 URL) and `POST /tenants/:id/erase` (tombstone + cascade delete, including trace payloads) as first-class endpoints, not an afterthought.

**Secrets/Credential Rotation** â€” Vault design covers encryption-at-rest and `validateCredentials()`/`healthCheck()`, but never rotation â€” only "rotate any keys pasted into notes files" (design.md Â§11). Static long-lived tokens are a major exposure vector (median breach detection: 204 days â€” [Groundcover](https://www.groundcover.com/learn/security/secret-rotation-how-it-works-challenges-best-practices)). Add a rotation reminder/expiry field on `credentials` plus a forced-rotation health-check failure path; for the master encryption key itself, adopt envelope encryption via a managed KMS (AWS KMS / **HashiCorp Vault** auto-rotation) instead of a static `CREDENTIALS_MASTER_KEY` env var ([HCP Vault rotation](https://developer.hashicorp.com/hcp/docs/vault-secrets/auto-rotation)).

**Health/Readiness/Liveness Checks** â€” No `/healthz`/`/readyz` endpoints for the Fastify API, and no worker-side check for BullMQ/Redis/Postgres connectivity. Without these, a crashed worker or exhausted DB pool fails silently until a job queue backs up. Best practice: liveness = process-alive only, readiness = dependency checks (Postgres, Redis, BullMQ queue depth) ([oneuptime Node.js pattern](https://oneuptime.com/blog/post/2026-01-06-nodejs-health-checks-kubernetes/view), [BullMQ worker checks](https://oneuptime.com/blog/post/2026-01-21-bullmq-worker-health-checks/view)). Cheap tool: **`@godaddy/terminus`** or a manual Fastify plugin pinging `db.execute('select 1')` + `redis.ping()` + BullMQ `queue.getJobCounts()`.

**Drizzle Migration Strategy** â€” Plan says only "run first migration" (Phase 0.2); no CI migration gate, no rollback plan, no rule against unsafe operations. Drizzle wraps migrations in transactions by default but `CONCURRENTLY` statements must be split into separate non-transactional files, and unsafe defaults (`DEFAULT now()`) trigger full-table rewrites ([Drizzle zero-downtime guide](https://dev.to/whoffagents/zero-downtime-postgres-migrations-with-drizzle-orm-22ga)). Recommendation: enforce `drizzle-kit generate` + manual SQL review before `migrate` in prod (never `push` outside dev), and a hard rule of additive-only changes per deploy (add column â†’ backfill â†’ deploy code â†’ drop old column, never combined).

**PII Handling Policy** â€” Only DM-specific redaction exists; there's no tenant-wide PII classification (what counts as PII across `engagement_items`, `intel_snapshots` scraped from public profiles, tenant customer lists) or retention schedule distinct from the "90 days for traces, forever for events" rule, which as written would retain PII indefinitely in `system_events`/`audit_log`. Best practice is tiering data (customer PII vs. internal telemetry) with different retention and deletion-on-request rules ([Forcepoint PII classification](https://www.forcepoint.com/blog/insights/pii-data-classification), [SaaS retention policy](https://www.buildmvpfast.com/blog/data-retention-policy-saas-startup-guide-2026)). Concrete: add a `contains_pii` flag + retention override at the row/category level in `system_events`, and a scheduled job that anonymizes/purges PII fields past the tenant's configured retention window independent of the audit log's "forever" default.
