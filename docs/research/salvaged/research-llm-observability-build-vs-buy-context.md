# Research LLM observability build-vs-buy context

## Research Findings: LLM Observability Build vs. Buy Landscape

### 1. Community consensus on comparisons (Langfuse vs Helicone vs Braintrust vs others)
- **Langfuse vs Helicone**: Helicone = proxy-based (Cloudflare Workers/ClickHouse/Kafka), one-line integration, built-in caching, best for quick setup and cost visibility. Langfuse = SDK-first, single Postgres-centric architecture, deeper tracing for complex/multi-step workflows. Rough consensus: Helicone for teams under ~$30K/mo LLM spend with simple chains; Langfuse once agents get complex or self-host/compliance matters. Some teams run both together (Helicone as gateway, Langfuse for tracing/evals). Sources: [helicone.ai/blog/best-langfuse-alternatives](https://www.helicone.ai/blog/best-langfuse-alternatives), [buildmvpfast.com comparison](https://www.buildmvpfast.com/blog/llm-observability-stack-langfuse-helicone-portkey-2026), [particula.tech comparison](https://particula.tech/blog/helicone-vs-langfuse-langsmith-llm-observability)
- **Langfuse vs Braintrust**: Langfuse = open-source, "building blocks" philosophy, requires self-managing Postgres/ClickHouse/Redis/S3 at scale; Braintrust = proprietary, managed, "batteries-included" with strong CI/CD eval-gating (blocks merges on quality regression). Source: [braintrust.dev/articles/langfuse-vs-braintrust](https://www.braintrust.dev/articles/langfuse-vs-braintrust)
- **OpenLLMetry/Traceloop**: Positioned as the pick when instrumentation portability (OTel-based, vendor-neutral) matters more than backend choice. Source: search results citing Braintrust/Firecrawl comparisons, e.g. [firecrawl.dev/blog/best-llm-observability-tools](https://www.firecrawl.dev/blog/best-llm-observability-tools)
- **Build vs buy framing**: Multiple 2025-2026 articles converge on "buy or use OSS self-host" as default; hidden build costs cited include span-schema churn per agent framework, evaluator engineering (months of work), and ongoing maintenance debt. Build is recommended only for air-gapped/strict data-residency cases. Sources: [futureagi.com build-buy-llm-observability-2025](https://futureagi.com/blog/build-buy-llm-observability-2025/), [helicone.ai/blog/buy-vs-build-llm-observability](https://www.helicone.ai/blog/buy-vs-build-llm-observability)

### 2. Langfuse license change — CONFIRMED, critical fact-check
Langfuse did **not** move between MIT and a different license overall; it operates (and has long operated) an **open-core dual-license model**, and that boundary **shifted in June 2025**:
- Core repo: MIT ("MIT Expat") for everything **except** code under `ee/`, `web/src/ee/`, `worker/src/ee/` directories, which carry a separate proprietary Enterprise License (`ee/LICENSE`). Source: [github.com/langfuse/langfuse/blob/main/LICENSE](https://github.com/langfuse/langfuse/blob/main/LICENSE)
- **June 4, 2025 change**: Langfuse open-sourced features that were *previously* commercial-only — managed LLM-as-a-judge evals, annotation queues, prompt experiments, and the playground — moving them to MIT. What remains commercial after this shrank to narrow "Enterprise Security and Platform" features: SCIM, Audit Logs, Data Retention Policies, Project-level RBAC, protected prompt labels, server-side data masking, UI customization, org management API. Regular SSO remains MIT. Source: [langfuse.com/blog/2025-06-04-open-sourcing-langfuse-product](https://langfuse.com/blog/2025-06-04-open-sourcing-langfuse-product) (also covered at [dev.to/clemra](https://dev.to/clemra/all-langfuse-product-features-now-free-open-source-4140))
- **Community confusion existed**: A GitHub discussion shows users were unclear whether free self-hosted Langfuse was commercially usable, and that license files referenced an "ee license that doesn't seem to exist" in visible form — maintainers clarified core = MIT, commercial use permitted. Source: [github.com/orgs/langfuse/discussions/5002](https://github.com/orgs/langfuse/discussions/5002)

### 3. Custom Postgres-based cost/trace tracking — is it reasonable?
Treated as a legitimate, common pattern, often paired with (not replacing) a dedicated tool: e.g., LiteLLM gateway writes spend records to Postgres keyed by `tenant_id`/model/API key (~20 lines of code), while Langfuse runs alongside for trace/quality monitoring. One engineering blog frames it as: custom Postgres tables give "lean, cost-focused attribution," dedicated tools give "immediate dashboards... at the cost of less direct control" — the choice hinges on whether the team wants to avoid running more infrastructure, not on technical capability. Source: [particula.tech/blog/per-tenant-llm-cost-attribution-multi-tenant-saas](https://particula.tech/blog/per-tenant-llm-cost-attribution-multi-tenant-saas)

### 4. Other self-hostable tools (brief)
- **Arize Phoenix**: Elastic License 2.0 (source-available, not OSI-approved), free self-host.
- **LangSmith**: closed source; self-hosting is a paid tier.
- **W&B Weave**: closed source; cloud + on-prem enterprise.
- **Langtrace**: open source, SOC 2 Type II.
Source: [medium.com/@kanerika LLMOps comparison](https://medium.com/@kanerika/llmops-observability-langsmith-vs-arize-vs-langfuse-vs-w-b-f1baeabd1bbf), [laminar.sh/article/arize-phoenix-alternatives-2026](https://laminar.sh/article/arize-phoenix-alternatives-2026)

### 5. What a bare Postgres table lacks vs. dedicated tools
Consensus across sources: the main value-add of tools like Langfuse/Helicone/Braintrust is **not** raw storage (a table can hold tokens/cost/latency/tenant_id fine) — it's the **visual trace tree** for stepping through multi-span agent runs to pinpoint failures "in minutes instead of hours," plus pre-built dashboards, prompt-versioning UI, and eval tooling. Source: [helicone.ai/blog/buy-vs-build-llm-observability](https://www.helicone.ai/blog/buy-vs-build-llm-observability)
