# Research opinions on MCP for backend automation

## Research Findings: Is MCP the Right Abstraction for a Deterministic Backend Publish-Worker?

### 1. What MCP's own spec/docs say it's FOR (establishes the baseline you're contrasting against)

- **MCP is architected around dynamic, model-driven tool selection, not a fixed script** — [modelcontextprotocol.io/docs/learn/architecture](https://modelcontextprotocol.io/docs/learn/architecture) — Official docs: "MCP clients will use the `*/list` methods to discover available primitives... This design allows listings to be dynamic," and in the worked example: *"When the language model decides to use a tool during a conversation, the AI application intercepts the tool call, routes it to the appropriate MCP server, executes it, and returns the results back to the LLM."* The host discovers tools at runtime via `tools/list` and the **model** decides when/whether to call `tools/call` — no dated version, current spec (2025-06-18/2025-11-25 protocol revisions referenced).
- **Per-tool human consent is a stated core design principle** — [modelcontextprotocol.io/specification/latest](https://modelcontextprotocol.io/specification/latest) — "Tools represent arbitrary code execution and must be treated with appropriate caution... **Hosts must obtain explicit user consent before invoking any tool**. Users should understand what each tool does before authorizing its use." This models a *per-step* human-in-the-loop pattern, distinct from a single upfront human approval gating an entire fixed sequence.
- **Original announcement framed MCP as a data/context-connection standard, not an automation-execution standard** — [Anthropic, "Introducing the Model Context Protocol"](https://www.anthropic.com/news/model-context-protocol), Nov 25, 2024 — "a new standard for connecting AI assistants to the systems where data lives... replacing fragmented integrations with a single protocol." Framed around giving models *context*, not scripting deterministic side-effecting workflows.
- **Anthropic itself has since published guidance that direct MCP tool-calling degrades at scale and recommends code execution instead** — [Anthropic Engineering, "Code execution with MCP: building more efficient AI agents"](https://www.anthropic.com/engineering/code-execution-with-mcp), Nov 4, 2025 — states that with many tools/large intermediate results, "tool definitions occupy more context window space, increasing response time and costs," and demonstrates a case going "from 150,000 tokens to 2,000 tokens." Notably this is Anthropic *itself* pointing away from raw per-step MCP tool-calling for anything resembling a fixed pipeline, toward writing code that orchestrates the calls instead.

### 2. Direct engineering opinions: MCP is overkill/wrong fit for deterministic, single-consumer, fixed-sequence automation

- **"If you only have one consumer, MCP is a liability"** — [TheProdSDE, dev.to, "MCP for Backend Engineers: When to Use It (and When to Skip It)"](https://dev.to/theprodsde/mcp-for-backend-engineers-when-to-use-it-and-when-to-skip-it-l3g), Apr 21, 2026 — "MCP represents a scaling decision, not a starting point... Plain tool calling always. Get the idea working first." Also frames the MCP payoff as solving "three copies of the same integration, each slightly wrong" across *multiple teams/hosts* — not applicable when one team owns both sides. Directly states single-consumer MCP costs "~1–2 extra days of setup for zero immediate benefit."
- **"Skip MCP for low-latency synchronous workflows... Skip MCP if you have a single model with three or fewer tools"** — [Ranjan Kumar, "Model Context Protocol (MCP): Architecture, Tradeoffs, and Production Realities"](https://ranjankumar.in/model-context-protocol-mcp-architecture-tradeoffs-and-production-realities), Feb 20, 2026 — "MCP is not a framework. It does not orchestrate agents. It does not manage conversation state." He explicitly justifies MCP's overhead as compensating for *model* nondeterminism: "Models hallucinate. They call tools they were not supposed to have access to... MCP adds operational overhead justified only when governance, auditability, and capability negotiation provide sufficient value" — i.e., value that assumes an LLM is actively choosing actions, which a fixed publish script doesn't need.
- **MCP is explicitly "not the right tool" for deterministic workflows** — WebSearch aggregate result surfaced from multiple sources on "MCP server backend automation overkill": "MCP is overkill when workflows anticipate the same outcome each time, as it's not intended for deterministic workflows that don't involve AI agents... ninety-five percent of MCPs fall into a category where they're slower, more complex versions of tools that already work perfectly." (Aggregated search-engine synthesis rather than one single article; treat as directional signal, not a single quotable source.)
- **"MCP is not an API. Tools are not agents"** — [Jim Clark, "You are Doing MCP Wrong: 3 Big Misconceptions"](https://www.docker.com/blog/mcp-misconceptions-tools-agents-not-api/), Docker blog, Sep 3, 2025 — "MCP is a model-facing protocol designed for LLM tool use, intent mediation, and context exchange. It does not replace RPC... its purpose is to make [APIs/RPC] safely and effectively usable by non-deterministic agents." Draws exactly your distinction: "**Non-deterministic layer:** model planning, tool choice, re-planning... **Deterministic layer:** tool execution, input validation, idempotency... **MCP as the seam** [between them]." His prescribed pattern for something like your publish-worker: keep the deterministic API/automation as-is, and only wrap it in MCP tool definitions if and when an LLM upstream needs to *reason about whether/when* to invoke it — not to drive the clicking itself.
- **"MCPs... are structured and predictable... no reasoning loops or decision making" vs Agents which "decide, step by step"** — [Maximiliano Gialleonardo, Ensolvers, "MCPs vs Agents"](https://www.ensolvers.com/post/mcps-vs-agents-how-are-they-different-when-to-use-each-and-how-to-confine-them), Sep 25, 2025 — recommends MCP for "structured and repetitive operations" needing "bulletproof execution," and agents/reasoning loops only where "context is more ambiguous." This is closer to a defense of using MCP-*style* interfaces even for deterministic work, but still concedes MCP itself doesn't add decision-making — the reasoning has to live elsewhere (which in your architecture is upstream, in the caption-drafting LLM, not in the browser worker).
- **Most complaints attributed to MCP are actually generic LLM/context problems, not protocol design flaws** (dissenting/contrarian take) — [Adam Gospodarczyk(?), Epic AI, "Why the Model Context Protocol Does Not Work"](https://www.epicai.pro/why-the-model-context-protocol-does-not-work-hgsz5) (byline as extracted; publish date not shown on page) — "there is very little mention of Model Context Protocol, and it isn't a mistake because most of the issues arise outside of it... even if it were flawless, it wouldn't solve the issues we face every day." This is a **dissenting** voice: doesn't argue MCP is the wrong abstraction for automation per se, argues the whole discourse over-targets MCP when the real issues are LLM context/hallucination/cost, which would exist regardless of transport.
- **Enterprises reject MCP's implicit trust in model behavior for production execution** — [Alex Salazar (CEO, Arcade.dev), Forbes Business Council, "Why MCP Alone Isn't Enough To Get Agents Into Production"](https://www.forbes.com/councils/forbesbusinesscouncil/2026/07/08/why-mcp-alone-isnt-enough-to-get-agents-into-production/), Jul 8, 2026 — "Most MCP implementations today rely on the model to behave correctly," which security teams reject; calls for an execution/authorization layer independent of MCP for auditable, attributable actions — relevant to a publish-worker needing hard guarantees (approval gate, tenant scoping) that can't be left to "the model behaving correctly."

### 3. Browser-automation-specific findings (playwright-mcp, Chrome DevTools MCP)

- **Token/context overhead of browser MCP servers is heavy relative to plain scripting** — [Mario Zechner, "What if you don't need MCP at all?"](https://mariozechner.at/posts/2025-11-02-what-if-you-dont-need-mcp/), Nov 2, 2025 — measured: Playwright MCP "13.7k tokens (6.8% of Claude's context)," Chrome DevTools MCP "18.0k tokens (9.0%)," versus a minimal bash-based tool README at "225 tokens." Argues for composable file-based/code tools over big generic MCP servers, especially since "the agent can decide to save [output] to a file for later processing" instead of routing everything through the model's context — a much closer fit for a fixed pipeline than a chatty per-action MCP loop.
- **Playwright MCP's own guidance: it's for exploration/authoring, not the deterministic runtime artifact** — Search synthesis from Playwright/testing-vendor sources (testcollab.com, currents.dev, testquality.com) converges on: "Playwright MCP accelerates exploration and test authoring; committed Playwright Test code remains the durable regression artifact... The MCP server is not needed at runtime" — i.e., even in Playwright's own ecosystem, MCP is positioned as a build-time/authoring aid for a human or agent iterating, and the **deterministic, repeatable execution path deliberately does not go through MCP at runtime.** This maps directly onto your case: your compose→attach→post sequence is the "committed spec," not something that needs live LLM tool selection each run.
- **HN: LLM-driven clicking is not reliably reproducible the way a script needs to be** — [HN: "Show HN: Browser MCP", item 43613194](https://news.ycombinator.com/item?id=43613194), Apr 2025 — commenter **thenaturalist** reported the first automation run worked "almost perfectly" but subsequent runs failed with click-detection errors; **throwaway314155** called this "luck," saying LLM-driven browser automation lacks "statistical reliability for repeated tasks" ("a good dice roll on your first turn"); **weq**: *"you can't just click on something in a script, you need to reliably click on something"* — traditional automation needs explicit fallback logic LLMs don't inherently provide. This is arguably the single most on-point primary-source quote for your report: it's engineers directly comparing LLM-mediated clicking against scripted determinism and picking scripted determinism for reliability.
- **HN: building a non-MCP alternative specifically to avoid per-action context overhead** — [HN: "Show HN: Playwright Skill for Claude Code – Less context than playwright-MCP", item 45642911](https://news.ycombinator.com/item?id=45642911) — author **syntax-sherlock**: built the skill because he "got tired of playwright-mcp eating through Claude's 200K token limit," replacing "sending accessibility tree snapshots on every action" with generated Playwright code returning only screenshots/console output. Commenter **yomismoaqui**: tried to get Playwright MCP to do a full-page screenshot and "it couldn't do it... I just told Claude Code to write a Playwright JS script to do that and it worked at the first try" — direct anecdote of MCP's tool-call abstraction underperforming a plain script for a concrete browser action.
- **HN: skepticism about whether MCP wrappers add real value vs. an LLM that already knows the underlying API** — same thread, **simonw**: raises whether specialized MCP tools genuinely improve on raw prompting, or whether "you usually just have to try it out" — general skepticism about MCP as a value-add layer when the model can already reason about a library/API directly.
- **HN: Browser MCP vs Playwright — selector fabrication problem** — [HN item 43613803](https://news.ycombinator.com/item?id=43613803), Apr 2025 — creator **namuorg** noted Playwright MCP "makes up" CSS selectors "rather than reading the page" in some flows (a determinism/reliability complaint specific to LLM-driven selector generation, the opposite of what a fixed compose→attach→post script needs); **behnamoh** dismissed the space as wheel-reinvention citing "nine prior browser-control projects spanning two years." Overall thread sentiment leaned positive on MCP for genuinely agentic use cases, with cost ("burned through $25 in 3 hours") as the main complaint rather than an architectural objection.

### 4. Overhead/latency, more generally

- **Round-trip and context-token accumulation across sequential tool calls** — aggregated from multiple engineering explainers (dev.to "Performance Tuning MCP Integrations," Medium/ByteBridge "MCP vs Traditional API Calls in Production," Anthropic's code-execution post) — recurring figures: "~50–100ms serialization overhead" per call, invisible for one action but compounding "for an agent looping over 50 files" or making many sequential calls; "MCP adds either a process hop (stdio) or a network hop (remote HTTP)... for an agent making 10 sequential calls with 50ms RTT, that's 500ms of overhead before tool execution even begins." For your 6–7 step deterministic flow (login → navigate → type → attach → click → confirm) this is a real, if modest, constant-factor tax with no corresponding benefit if nothing is actually being decided at each step.
- **Anthropic's own quantification of tool-definition/result overhead at scale** — see §1 above (150k→2k tokens example) — this is Anthropic's internal justification for *not* wanting agents to route every step's output back through raw context, reinforcing that pure per-action MCP tool-calling is something even MCP's authors consider inefficient once a workflow is long/fixed enough to just be code.

### 5. Consensus vs. dissent, as observed

**Rough consensus across independent sources:** MCP's value proposition assumes (a) an LLM is actively choosing which tool to invoke and in what order at runtime, (b) there are multiple tools/consumers/teams where standardization avoids duplicated integration work, and (c) per-tool human consent/governance semantics are desired. None of these premises hold for a fixed, pre-scripted publish sequence run by one service after a single upstream human approval — multiple named sources (TheProdSDE, Ranjan Kumar, Docker/Jim Clark, Zechner) independently converge on "skip MCP" language for exactly this shape of problem (single consumer, deterministic sequence, low tool count, latency-sensitive). The recommended pattern from several of these (Docker blog especially) is: keep the deterministic automation as ordinary code/API calls; only put an MCP layer in front of it if/when a *different, LLM-driven* consumer later needs to reason about whether to invoke it.

**Dissenting/complicating views:** (1) epicai.pro argues the anti-MCP discourse over-blames the protocol for problems that are really generic LLM/context issues — a caution against overstating "MCP is bad," though it doesn't actually argue MCP *is* right for deterministic backend automation. (2) Docker/Ensolvers-style "wrap your API in MCP anyway" advocates note there can be latent value in exposing even deterministic operations behind MCP-shaped tool definitions purely for future optionality/governance/observability — this is the strongest counter-argument you'd need to address in the report, but it's an "optionality" argument, not a claim that MCP is a *better fit* for the runtime execution itself.
