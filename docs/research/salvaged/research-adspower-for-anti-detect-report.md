# AdsPower — Research Report

## 1. What it is and how it works

AdsPower is an "anti-detect browser" (also called anti-fingerprint / antidetect browser) whose core technical claim is that it lets one operator run many isolated browser "profiles" that each present a unique, internally-consistent device/browser fingerprint to websites, so that platforms cannot link the profiles back to a single real operator. (https://www.adspower.com/, https://www.adspower.com/glossary/docs/antidetect-browser)

- **Fingerprint isolation/spoofing**: Each profile is described as "a separate virtual environment — with its own user agent, timezone, language, operating system, and hardware fingerprint," with each profile's cookies, cache, and local storage kept separate so that "social media and marketplace platforms are far less likely to link" the accounts. AdsPower advertises "over 20 options, 50+ customizable parameters" covering IP, language, resolution, timezone, cookies, geolocation, fonts, proxy, DNS, Canvas and WebGL. (https://www.adspower.com/, per WebFetch synthesis of homepage copy)
- **Execution model — local, not cloud**: The product runs client-side: users "Launch AdsPower and Click New Profile," then "Save the profile and open it when you need," i.e., the browser itself executes on the user's own machine rather than in a remote/cloud browser session; profile configs are synced/managed through AdsPower's own client + a "Synchronizer" feature rather than the whole browser being cloud-hosted. (https://www.adspower.com/antidetect-browser)
- **Two browser engines**: AdsPower ships two browser cores — "SunBrowser" (Chromium-based) and "FlowerBrowser" (Firefox-based) — which the vendor says get frequent kernel updates ("14 major kernel updates" in 2025 per their own blog) specifically so the declared browser version matches the real underlying engine and doesn't create a fingerprint mismatch. (https://www.adspower.com/, https://www.adspower.com/blog/how-browser-fingerprinting-triggers-account-locks)
- **Detection-evasion detail at the network layer**: AdsPower's own blog explains platforms cross-check TLS/JA3 fingerprints, HTTP headers (User-Agent, Sec-CH-UA) and JS APIs (Canvas, WebGL, AudioContext) for internal consistency, and states "a single mismatch at any layer is enough to trigger a block or account lock" — AdsPower positions its dual-engine + "Native Mobile Simulation" (aligning iOS/Android user-agents, device parameters and TLS fingerprints with real devices) as the fix for this. (https://www.adspower.com/blog/how-browser-fingerprinting-triggers-account-locks)
- **Built-in "RPA"/automation**: The homepage advertises "Streamline your workflow with RPA, API integration, and Synchronizer" with "ready-to-use templates or customize your own automation tasks" — i.e., a no-code/low-code scripted automation layer inside the GUI product, separate from the programmatic API discussed below. (https://www.adspower.com/, https://www.adspower.com/antidetect-browser)

## 3. Programmatic API/SDK

AdsPower explicitly ships a full programmatic control surface, not just a GUI — this is a real API for headless/scripted control, and it is directly compatible with Selenium and Puppeteer via a local CDP endpoint.

- **"Local API"**: A local HTTP API running on the user's own machine, documented at https://localapi-doc-en.adspower.com/ with default endpoints `http://local.adspower.net:50325/` or `http://localhost:50325/`. It supports reading/writing profile configuration, and creating, opening, closing, and querying profiles programmatically. (https://localapi-doc-en.adspower.com/docs/Rdw7Iu, https://github.com/AdsPower/localAPI)
- **CDP / Puppeteer / Selenium compatibility — confirmed with specifics**: The "Open Browser" API call (https://localapi-doc-en.adspower.com/docs/FFMFMf) returns, among other fields:
  - `"selenium":"127.0.0.1:xxxx"` — a Selenium debugger address
  - `"puppeteer":"ws://127.0.0.1:xxxx/devtools/browser/xxxxxx"` — a WebSocket address whose path (`/devtools/browser/...`) is the standard Chrome DevTools Protocol (CDP) browser endpoint format, i.e., this is a genuine local CDP endpoint that Puppeteer (and any other CDP client) can attach to
  - `"debug_port"` and `"webdriver"` (path to a matching chromedriver executable)
  (https://localapi-doc-en.adspower.com/docs/FFMFMf)
- **Notable "cdp_mask" parameter**: The Open Browser call also accepts a `cdp_mask` parameter, documented purpose "to mask the cdp detection" (default value 1/on). This means AdsPower has a built-in feature specifically designed to hide from websites the telltale signals that a browser session is being driven by CDP-based automation (Selenium/Puppeteer/Playwright detection scripts commonly look for CDP artifacts) — i.e., it's not just an automation API, it's an automation API paired with anti-bot-detection evasion for that automation. (https://localapi-doc-en.adspower.com/docs/FFMFMf)
- **Access gating**: The docs state that only "purchased paid versions of AdsPower are granted permission to use the API" — the free tier does not include API access. (https://localapi-doc-en.adspower.com/docs/Rdw7Iu)
- Additional resources: a Postman collection covering "Browser Automation, Profile, Proxy, Member Management" (https://documenter.getpostman.com/view/45822952/2sB34hEzQH), a public GitHub client library (https://github.com/AdsPower/localAPI), and an official Medium walkthrough ("Introduction to AdsPower Local API," https://adspower.medium.com/introduction-to-adspower-local-api-298773e945fe).

## Bottom line relevant to the SaaS evaluation

For a tool built around one real human per account, with human-approved posts, AdsPower's core value proposition (fingerprint spoofing across many isolated profiles to defeat platform multi-account/bot detection, plus a `cdp_mask` feature to hide automation from anti-bot systems) is architecturally oriented toward exactly the multi-accounting/ban-evasion use case the target SaaS explicitly wants to avoid resembling. Its programmatic API is real and CDP/Puppeteer/Selenium-compatible (confirmed via https://localapi-doc-en.adspower.com/docs/FFMFMf), so it is technically capable of driving legitimate one-account automation too — but its pricing, docs, and marketing language are all built around the profile-scaling/detection-evasion model rather than a single-tenant "your own account, your own consent" model.
